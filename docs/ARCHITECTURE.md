## 1. Project Overview

### Vision
`claude-from-dotnet` es una guía práctica en forma de repositorio público de GitHub. Su objetivo es enseñar a desarrolladores .NET cómo integrar la API de Anthropic Claude en aplicaciones reales, sin tener que aprender Python ni cambiar de stack. Cada concepto se presenta como un ejemplo independiente, ejecutable de forma aislada con `dotnet run`, usando `HttpClient` directo y sin SDKs no oficiales que añadan magia oculta.

Es **producto y posicionamiento a la vez**: por un lado, una referencia técnica útil para la comunidad .NET hispanohablante; por otro, una prueba pública de que el autor (Sergio Ojeda Torres, dev .NET con ~3 años de experiencia) integra IA en software real de forma criteriosa, no a base de copiar tutoriales.

### Goals
- Convertir el repo de "coming soon" a 6 ejemplos funcionales en las próximas 4-6 semanas (uno por semana aproximadamente)
- Cada ejemplo debe correr con `dotnet run` después de copiar `.env.example` a `.env` y meter una API key — cero fricción
- El código debe ser legible para alguien que sabe C# pero nunca ha tocado un LLM por API
- Posicionar al autor como referencia .NET + IA aplicada en LinkedIn y búsquedas de Google

### Success Metrics
- 6 ejemplos completos publicados al final del mes 1
- ≥50 stars en GitHub al final del mes 3 (proxy de tracción real)
- Al menos 1 contacto entrante (recruiter o cliente) que mencione el repo como razón para escribir

## 2. Tech Stack

| Layer | Technology | Why |
|-------|-----------|-----|
| Runtime | .NET 8 (LTS) | LTS hasta nov 2026, ampliamente soportado en producción. .NET 9 ya está pero LTS gana en docs de referencia. |
| Lenguaje | C# 12 | Soporte completo en .NET 8, nullable references por defecto, primary constructors. |
| HTTP Client | `System.Net.Http.HttpClient` (built-in) | Decisión clave del repo: NO usar SDKs no oficiales. El lector aprende qué hace la API por debajo. |
| Serialización JSON | `System.Text.Json` | Built-in. Cero dependencias extra. Soporta JsonSerializerContext para source generators si se quiere. |
| Streaming SSE | Lectura manual de `Stream` línea a línea | Demuestra cómo funciona SSE sin abstracciones. Anthropic devuelve `event: ...` + `data: ...`. |
| Config / Secrets | `dotenv.net` (NuGet, MIT) | Carga `.env` en `Environment` variables. Una sola dependencia, ampliamente usada. Alternativa: `DotNetEnv`. |
| Logging | `Microsoft.Extensions.Logging` con consola | Built-in. Suficiente para ejemplos didácticos. |
| Tipo de proyecto | Console app por ejemplo | Cada ejemplo es un `dotnet new console`. Self-contained, ejecutable directamente. |
| Tests | xUnit (opcional, solo en `shared/`) | Solo si se extrae un cliente compartido. La mayoría de ejemplos no llevan tests para no contaminar la pedagogía. |
| Solution | Un único `.sln` en la raíz | Permite abrir el repo entero en VS Code / Rider / VS y navegar entre ejemplos. |
| CI | GitHub Actions — build matrix | Verifica que cada `.csproj` compila en cada push. No tests obligatorios. |
| Package Manager | NuGet (vía `dotnet`) | Estándar .NET. |
| Hosting | GitHub público (ya existe) | El "deploy" es publicar en GitHub. No hay servicio que desplegar. |

**Decisiones que NO tomamos y por qué:**
- **No Anthropic SDK no oficial** — Los lectores aprenden la API real, no una capa de abstracción.
- **No ASP.NET / Minimal API en los primeros ejemplos** — Añade complejidad innecesaria. El ejemplo MCP server (futuro) sí lo necesitará.
- **No Polly para reintentos hasta el ejemplo 6+** — La primera vez se ve manualmente con un `try/catch` + delay, luego se introduce Polly como mejora.
- **No Refit / RestSharp** — Mismo motivo que el SDK: tapa la API que se quiere enseñar.

## 4. Data Model

No aplica como base de datos. Los "datos" del proyecto son las request/response shapes de la API de Anthropic, que se modelan con `record` de C# 12 + atributos de `System.Text.Json`.

### Modelos principales (compartidos entre ejemplos)

**`Message`** — bloque de conversación
| Campo | Tipo | Notas |
|-------|------|-------|
| Role | string | "user" \| "assistant" |
| Content | string \| ContentBlock[] | Puede ser texto plano o lista de bloques (texto, tool_use, tool_result, etc.) |

**`MessagesRequest`** — request a `POST /v1/messages`
| Campo | Tipo | Notas |
|-------|------|-------|
| Model | string | Ej: `claude-sonnet-4-6` |
| MaxTokens | int | Obligatorio. Recomendado 1024 para ejemplos. |
| Messages | Message[] | Historial de la conversación |
| System | string? | Prompt de sistema opcional |
| Tools | Tool[]? | Solo en ejemplo 04+ |
| Stream | bool? | true en ejemplo 02 |

**`MessagesResponse`** — response completa (no streaming)
| Campo | Tipo | Notas |
|-------|------|-------|
| Id | string | ID del mensaje |
| Type | string | Siempre "message" |
| Role | string | Siempre "assistant" |
| Content | ContentBlock[] | Texto + tool calls |
| Model | string | El modelo que respondió |
| StopReason | string | "end_turn" \| "tool_use" \| "max_tokens" |
| Usage | Usage | input_tokens, output_tokens |

**`StreamEvent`** — para ejemplo 02
| Campo | Tipo | Notas |
|-------|------|-------|
| Type | string | "message_start" \| "content_block_delta" \| "message_stop" \| ... |
| Index | int? | Índice del bloque de contenido |
| Delta | object? | Contenido incremental |

### Esquemas JSON enviados a la API

Todos siguen la documentación oficial de Anthropic en `docs.claude.com/en/api/messages`. El repo incluye un resumen propio en `docs/api-reference-resumen.md` para que el lector no tenga que saltar a la doc oficial constantemente.

## 5. API Design

No hay API que diseñar — este repo es **consumidor** de la API de Anthropic. La sección equivalente es **"qué endpoints de Anthropic usamos y cómo"**:

| Ejemplo | Endpoint Anthropic | Método | Descripción |
|---------|--------------------|--------|-------------|
| 01-chat-basico | `/v1/messages` | POST | Petición simple, respuesta completa |
| 02-streaming | `/v1/messages` con `stream: true` | POST | SSE, parseo manual |
| 03-structured-output | `/v1/messages` con JSON schema en system prompt + `tool_choice` truco | POST | Estructurar la salida como JSON deserializable |
| 04-tool-use | `/v1/messages` con `tools: [...]` | POST | Loop: pedimos → tool_use → ejecutamos → tool_result → seguimos |
| 05-mcp | `/v1/messages` con tools generados desde MCP server | POST | MCP client en C# que descubre tools de un server stdio |
| 06-rag | `/v1/messages` + endpoint de embeddings externo | POST | Embeddings (OpenAI/Cohere) + búsqueda → contexto en system prompt |

### Headers obligatorios en cada llamada
```
x-api-key: $ANTHROPIC_API_KEY
anthropic-version: 2023-06-01
content-type: application/json
```

### Manejo de errores (patrón común a todos los ejemplos)
- 400 → request mal formada (problema del developer, mostrar respuesta exacta)
- 401 → API key inválida (mensaje claro: revisar `.env`)
- 429 → rate limit (esperar `retry-after` segundos y reintentar)
- 500/529 → error del lado de Anthropic (reintentar con backoff exponencial)
- Otros → log + propagar

Se introduce reintento manual en el ejemplo 02 y se reescribe con Polly en un anexo del ejemplo 06.

## 7. Design System

### Branding del repo (solo afecta a README.md, docs/, y commit messages)

- **Lenguaje:** Español. Code comments en español también. Mensajes de commit en español.
- **Tono:** directo, no académico. Como un colega contándotelo por Slack.
- **Sin emojis** en código. Permitidos en READMEs si aportan (`✅`, `🚧`, `📦`) — no decorativos.
- **Sin "hello world" en los ejemplos** — siempre algo mínimamente útil (resumir un texto, extraer datos, etc.).

### Convenciones de código C#
- **Indentación:** 4 espacios.
- **Nullable reference types:** ENABLED en todos los `.csproj`.
- **File-scoped namespaces:** sí.
- **Naming:** PascalCase para tipos y métodos, camelCase para variables locales, `_camelCase` para campos privados.
- **`var` siempre** que el tipo sea obvio del RHS.
- **Records preferidos** sobre clases para DTOs de API.
- **Async/await siempre.** Nada de `.Result` ni `.Wait()`.

### Convenciones de carpetas dentro de cada ejemplo
- `Program.cs` en la raíz del ejemplo, con `Main` corto que delega a métodos.
- Modelos en `Models/`, uno por archivo.
- Lógica auxiliar en carpetas con nombre descriptivo (`Sse/`, `Tools/`, `Embeddings/`).
- README al lado del código, no en `docs/`.

## 8. Authentication & Authorization

### Para el repo en sí
Repo público. No hay auth. Los lectores hacen fork o clonan.

### Para los ejemplos al ejecutarse
Cada ejemplo necesita una API key de Anthropic, leída de `ANTHROPIC_API_KEY` vía `dotenv.net`. El ejemplo 06 además necesita una API key del proveedor de embeddings (OpenAI o Cohere).

### Patrón de carga de secretos (común a todos los ejemplos)
```csharp
DotNetEnv.Env.Load(); // Carga .env en environment variables
var apiKey = Environment.GetEnvironmentVariable("ANTHROPIC_API_KEY")
    ?? throw new InvalidOperationException("Falta ANTHROPIC_API_KEY en .env");
```

### Reglas no negociables sobre secretos
1. **`.env` NUNCA en git.** Está en `.gitignore` raíz.
2. **`.env.example` SIEMPRE presente** en cada ejemplo, con la lista de variables necesarias y valor vacío.
3. **NUNCA hardcodear keys.** Ni en comentarios, ni en strings, ni en tests.
4. **Si Git detecta una key en un commit** → revertir, rotar la key en console.anthropic.com, force-push si es necesario.

## 9. Build Order

> Este es el orden recomendado para que un Claude Code (o un humano) construya el repo completo desde cero. Cada paso es un commit autónomo y debe dejar el repo en estado verde (`dotnet build` pasa).

### **Step 1: Scaffolding inicial del repo**
- Crear `ClaudeFromDotnet.sln` en la raíz: `dotnet new sln -n ClaudeFromDotnet`
- Crear `.gitignore` con plantilla `.NET` de GitHub (`bin/`, `obj/`, `.vs/`, `.idea/`, `*.user`) + `.env` + `*.local.json`
- Crear `.editorconfig` con reglas C# estándar (4 spaces, LF, charset UTF-8)
- Crear `.env.example` en la raíz con `ANTHROPIC_API_KEY=` (sin valor)
- Crear carpetas vacías: `examples/`, `docs/`, `.github/workflows/`
- Commit: `chore: scaffolding inicial del repo`

### **Step 2: Documentación de setup**
- Crear `docs/00-setup.md` explicando:
  - Cómo crear cuenta en console.anthropic.com y generar API key
  - Cómo instalar .NET 8 SDK en Windows / macOS / Linux
  - Cómo clonar el repo y abrirlo en VS Code (con extensión C# Dev Kit) / Rider / VS
  - Cómo correr el primer ejemplo
- Crear `docs/glosario.md` con términos clave (token, modelo, system prompt, streaming, tool, MCP, RAG, embeddings, vector similarity)
- Crear `docs/api-reference-resumen.md` con un resumen propio de los 2-3 endpoints de Anthropic que se usan
- Commit: `docs: setup, glosario y resumen API`

### **Step 3: GitHub Actions de build**
- Crear `.github/workflows/build.yml`:
  - Trigger: push a `main`, pull_request
  - Matrix: ubuntu-latest (suficiente; añadir windows-latest solo si hay quejas)
  - Pasos: setup .NET 8 → `dotnet restore` → `dotnet build --no-restore -c Release`
- Commit: `ci: build de toda la solución en GitHub Actions`

### **Step 4: Ejemplo 01 — Chat básico**
- `cd examples/` → `dotnet new console -n 01-chat-basico`
- Añadir a la solution: `dotnet sln ../ClaudeFromDotnet.sln add examples/01-chat-basico/01-chat-basico.csproj`
- Habilitar nullable y file-scoped namespaces en `.csproj`
- `dotnet add package dotnet.env` (o `DotNetEnv`)
- Crear `Models/MessagesRequest.cs`, `Models/MessagesResponse.cs`, `Models/Message.cs` (records mínimos)
- Implementar `Program.cs`:
  - Carga `.env`
  - Construye `HttpClient` con `BaseAddress = https://api.anthropic.com`
  - Añade headers `x-api-key`, `anthropic-version: 2023-06-01`
  - Serializa request con `System.Text.Json`
  - POST a `/v1/messages` con `{"model": "claude-sonnet-4-6", "max_tokens": 1024, "messages": [{"role": "user", "content": "Hola, ¿quién eres?"}]}`
  - Deserializa respuesta
  - Imprime `response.Content[0].Text`
- Crear `.env.example` con `ANTHROPIC_API_KEY=`
- Crear `README.md` siguiendo la plantilla de Sección 6
- Verificar: `cd examples/01-chat-basico && cp .env.example .env` → meter key real → `dotnet run` → debe imprimir respuesta de Claude
- Commit: `feat(01): chat básico con HttpClient`

### **Step 5: Ejemplo 02 — Streaming**
- Mismo scaffold que ejemplo 01 pero con un parser SSE manual
- Crear `Sse/SseReader.cs`:
  - Recibe un `Stream` (de `HttpResponseMessage.Content.ReadAsStreamAsync()`)
  - Lee línea a línea con `StreamReader.ReadLineAsync()`
  - Parsea `event: <type>\ndata: <json>\n\n` → eventos tipados
- En `Program.cs`:
  - Mismo POST que ejemplo 01 pero con `stream: true` y `Accept: text/event-stream`
  - Pasa el stream a `SseReader`
  - Por cada evento de tipo `content_block_delta`, imprime el `delta.text` sin newline
- README explica qué es SSE y por qué se usa para streaming
- Commit: `feat(02): streaming de respuestas con SSE`

### **Step 6: Ejemplo 03 — Structured output**
- Pedir a Claude JSON estructurado usando system prompt con esquema explícito
- Crear `Schemas/PersonaSchema.cs` (record con `Nombre`, `Edad`, `Email`, etc.)
- En `Program.cs`:
  - System prompt: "Devuelve exactamente este JSON, sin texto extra: {esquema}"
  - User prompt: "Extrae datos de esta firma de email: ..."
  - Recibe respuesta, hace `JsonSerializer.Deserialize<Persona>(response.Content[0].Text)`
  - Imprime el record deserializado
- Nota en README sobre `tool_choice` como alternativa más robusta (mencionar, no implementar todavía — se ve en el 04)
- Commit: `feat(03): structured output con JSON schema`

### **Step 7: Ejemplo 04 — Tool use**
- Implementar el loop tool_use ↔ tool_result
- Crear `Tools/`:
  - `IClaudeTool.cs` (interfaz: `Name`, `Description`, `InputSchema`, `Execute(JsonNode input)`)
  - `GetWeatherTool.cs` (mock: devuelve "Soleado, 22°C" para cualquier ciudad)
  - `AddNumbersTool.cs` (suma dos números)
- En `Program.cs`:
  - Registra las tools, las serializa al campo `tools` de la request
  - Loop:
    1. POST mensajes a Claude
    2. Si `stop_reason == "tool_use"` → ejecutar cada `tool_use` block → añadir `tool_result` a la conversación → ir a 1
    3. Si `stop_reason == "end_turn"` → imprimir respuesta final
- README explica el loop con un diagrama ASCII
- Commit: `feat(04): tool use con loop manual`

### **Step 8: Ejemplo 05 — MCP**
- Implementación mínima de un MCP client en C# que se conecta vía stdio a un server externo
- Crear `McpClient/`:
  - `McpClient.cs` (lanza proceso del server, comunica vía stdin/stdout con JSON-RPC 2.0)
  - `McpProtocol.cs` (request/response shapes según spec MCP)
- En `Program.cs`:
  - Conecta al MCP server de referencia (`@modelcontextprotocol/server-filesystem` vía npx, por ejemplo)
  - Lista las tools que expone
  - Las convierte al formato `tools` de Anthropic
  - Reutiliza el loop del ejemplo 04 para que Claude pueda invocarlas
- README: tabla con MCP servers de referencia para probar (filesystem, github, postgres, etc.) + comando exacto para levantarlos
- Commit: `feat(05): integración MCP vía stdio`

### **Step 9: Ejemplo 06 — RAG mínimo**
- RAG completo pero sin dependencias pesadas
- Crear `data/` con 3-4 archivos `.md` de ejemplo (FAQs, política de empresa ficticia, manual técnico)
- Crear `Embeddings/OpenAiEmbeddingsClient.cs`:
  - HttpClient a `https://api.openai.com/v1/embeddings`
  - Modelo `text-embedding-3-small`
- Crear `VectorStore/InMemoryVectorStore.cs`:
  - Lista de `(float[] vector, string text, string source)`
  - Método `Search(query, topK)` que calcula similitud coseno
- En `Program.cs`:
  - Boot: indexa todos los `.md` de `data/` → embeddings → vector store
  - Loop: pregunta del user → embedding de la pregunta → top 3 chunks → system prompt con esos chunks → POST a `/v1/messages` → respuesta
- README explica embeddings, similitud coseno y cuándo NO usar RAG (cuando un long context window basta)
- Mencionar que en producción se usa una vector DB de verdad (pgvector, Qdrant, Pinecone)
- Commit: `feat(06): RAG mínimo con embeddings y vector store en memoria`

### **Step 10: Actualizar README raíz**
- Tras cada ejemplo, marcar la tabla de estado del README raíz como ✅
- Cuando los 6 estén ✅, añadir sección "Roadmap futuro" con ideas: streaming de tool calls, MCP server propio, caching, evaluaciones automáticas

### **Step 11: Polish y publicación**
- Pasar `dotnet format` a todo el repo
- Añadir badges en el README: build status, .NET version, license
- Crear un primer release en GitHub (`v0.1.0`) cuando los 6 ejemplos estén
- Compartir en LinkedIn con un post técnico explicando el repo (ver roadmap personal brand)

## 13. Testing Strategy

### Política general
**El código de ejemplos NO se testea.** La razón: tests añadirían ruido al código que se quiere enseñar. El "test" de cada ejemplo es: `dotnet run` y verificar la salida esperada (documentada en el README del ejemplo).

### Excepciones
- Si se introduce `shared/ClaudeFromDotnet.Shared/` (a partir del ejemplo 04+), esa librería SÍ lleva tests con xUnit.
- El parser SSE del ejemplo 02 puede llevar tests si se vuelve complejo (deserializar SSE manualmente da bugs sutiles).

### Verificación manual (lo que sustituye a tests automáticos)
Cada README de ejemplo lista:
- Comando para correrlo
- Output esperado (literal o regex)
- Errores comunes y cómo arreglarlos

### CI no falla por falta de tests
El workflow solo hace `dotnet build`. No corre tests porque no hay tests obligatorios.

```markdown
# claude-from-dotnet

Guía práctica de ejemplos C# / .NET para integrar la API de Anthropic Claude usando HttpClient directo, sin SDKs no oficiales.

## Commands

- `dotnet build` — Compila toda la solution
- `dotnet run --project examples/01-chat-basico` — Corre un ejemplo concreto
- `dotnet format` — Aplica el estilo de código del .editorconfig
- `dotnet sln add examples/NN-nuevo-ejemplo/NN-nuevo-ejemplo.csproj` — Añade un nuevo ejemplo a la solution

## Tech Stack

.NET 8 + C# 12 + HttpClient (built-in) + System.Text.Json (built-in) + dotenv.net (única dependencia común). Sin frameworks web. Cada ejemplo es una console app independiente.

## Architecture

### Directory Structure
- `examples/NN-nombre/` — Un ejemplo autónomo por carpeta. Cada uno tiene su `.csproj`, `Program.cs`, `Models/`, `.env.example`, `README.md`.
- `docs/` — Setup, glosario, resumen API. Documentación transversal.
- `shared/ClaudeFromDotnet.Shared/` — Solo si la duplicación entre ejemplos se vuelve insostenible (no antes del ejemplo 04).
- `.github/workflows/build.yml` — CI que verifica que toda la solution compila.

### Data Flow
1. Cada ejemplo carga `.env` con `DotNetEnv.Env.Load()` al arranque
2. Lee `ANTHROPIC_API_KEY` con `Environment.GetEnvironmentVariable`
3. Crea `HttpClient` con `BaseAddress = https://api.anthropic.com` y headers `x-api-key` + `anthropic-version`
4. Serializa request con `System.Text.Json`
5. POST a `/v1/messages`
6. Deserializa response (o lee stream SSE en el ejemplo 02)
7. Imprime resultado por consola

### Key Patterns
- **Cada ejemplo es totalmente self-contained.** Un lector puede copiar UNA carpeta de `examples/` fuera del repo y seguirá funcionando.
- **Records inmutables** para todos los DTOs de la API. Nada de clases con setters.
- **Nullable reference types habilitados** en todos los `.csproj`. Tratar warnings como errores.
- **HttpClient como singleton** dentro de cada Program.cs (no inyectado, son apps de un solo uso).
- **Async/await en todas las llamadas a la API.** Nunca `.Result` ni `.Wait()`.

## Code Organization Rules

1. **`Program.cs` corto.** `Main` no más de 30 líneas. Si crece, extraer métodos estáticos en el mismo archivo o clases en `Services/`.
2. **Un record por archivo.** Modelos en `Models/`. Cada record en su archivo.
3. **File-scoped namespaces.** `namespace ClaudeFromDotnet.E01ChatBasico;` al principio del archivo, sin llaves.
4. **No comments obvios.** Comentar el por qué, no el qué.
5. **README al lado del código.** Cada `examples/NN-*/` tiene su `README.md` con la plantilla estándar.

## Design System

No hay frontend. El "design system" son las reglas de estilo del código:
- Indentación: 4 espacios
- Line endings: LF
- Charset: UTF-8
- `var` cuando el tipo es obvio del RHS
- PascalCase tipos/métodos, camelCase locales, `_camelCase` campos privados

## Environment Variables

| Variable | Descripción |
|----------|-------------|
| `ANTHROPIC_API_KEY` | API key de Anthropic. Obligatoria en todos los ejemplos. |
| `ANTHROPIC_MODEL` | Modelo (opcional, default `claude-sonnet-4-6`). |
| `OPENAI_API_KEY` | Solo ejemplo 06 (embeddings). |

## Reglas No Negociables

1. **NUNCA un SDK no oficial de Anthropic.** Solo HttpClient directo. La pedagogía del repo depende de esto.
2. **NUNCA commitear `.env`.** Está en `.gitignore` raíz. `.env.example` sí, `.env` no.
3. **NUNCA hardcodear API keys.** Ni en strings, ni en comentarios, ni en tests. Si pasa, rotar la key inmediatamente en console.anthropic.com.
4. **Cada NuGet añadido necesita justificación** en el README del ejemplo. Si se puede hacer con built-in, se hace con built-in.
5. **README de cada ejemplo es obligatorio** y sigue la plantilla de Sección 6 del blueprint.
6. **El idioma del repo es español.** Code comments, READMEs y commit messages en español. Excepción: los nombres de tipos de la API de Anthropic se mantienen en inglés (`MessagesRequest`, `StopReason`, etc.).
```

## 16. Reglas No Negociables (proyecto entero)

1. **HttpClient directo siempre.** Cero SDKs no oficiales para Anthropic. La razón de existir del repo es enseñar la API real, no una capa de abstracción.
2. **Cada ejemplo es self-contained.** El lector copia una carpeta y funciona. No hay imports relativos a `shared/` en los primeros 3 ejemplos.
3. **`.env` nunca en git.** Solo `.env.example`. Si entra una key por accidente → rotar key, force-push, mea culpa pública si el repo era público.
4. **Nullable reference types ENABLED** en todos los `.csproj`. Sin excepciones.
5. **Una dependencia NuGet sin justificación = rechazado.** Built-in primero.
6. **Todo en español** (READMEs, comentarios, commits). Tipos de la API en inglés porque así los nombra Anthropic.
7. **CI verde antes de mergear cualquier cosa a `main`.** Si `dotnet build` falla en el workflow, no se mergea.
8. **El README raíz se actualiza al cerrar cada ejemplo.** La tabla de estado refleja la realidad.
9. **Async/await siempre.** Cero `.Result` o `.Wait()`. Si hay deadlock en una console app es bug del developer.
10. **Pedagogía sobre DRY** en los primeros 3 ejemplos. Es mejor duplicar código entre 01 y 02 que obligar al lector a saltar a `shared/` para entender qué pasa.
